## 一、工作目标

为电影管理程序新增对外的HTTP REST API接口，使移动端App、Web客户端或第三方工具可以通过HTTP协议访问电影、分类、标签、演员、收藏夹、播放历史等核心数据。

具体目标：

1. 使用Node.js原生 `http` 模块（不引入Express/Koa等额外依赖），按照 `design/api-design.md` 中定义的路由与响应规范实现HTTP服务，代码统一收敛到 `src/main/services/HttpService.js`。
2. 在 `config/settings.json` 中新增 `http` 配置节，控制服务的启用状态、监听地址与端口：
   ```jsonc
   "http": {
       "enabled": false,        // 默认关闭
       "listenAddress": "0.0.0.0",
       "listenPort": 8080
   }
   ```
3. 在主窗口设置弹窗（`index.html` 的 `settings-modal`）的"网络配置"Tab中新增HTTP服务面板，允许用户在不修改文件的情况下启用/禁用服务、修改监听地址与端口。
4. 必须满足：
   - 所有新增逻辑均覆盖单元测试，并执行通过；
   - 代码遵循CleanCode（命名清晰、单一职责、注释完备）；
   - 整理本工作日志（中文，存放在 `worklog/<timestamp>-worklog.md`）。

## 二、设计思路与关键决策

### 2.1 架构选择：协议适配层 + 路由表

`HttpService` 定位为**协议适配层**：不重复实现任何业务逻辑，只把HTTP请求翻译成对现有Service（`MovieService` / `BoxService` / `CategoryService` / `TagService` / `ActorService` / `MovieHistoryService`）的调用，再把结果包装成统一响应格式返还。

```
HTTP Request ──► HttpService（路由匹配） ──► 控制器handle*函数 ──► 调用现有Service ──► 统一JSON响应
```

这样做的好处：

- 不入侵任何现有Service，已有的GUI/IPC调用链不受影响；
- 设计文档中"附录B：与现有服务的映射关系"可以直接落地到每个控制器；
- 未来如果替换HTTP框架（如改用Express），仅需替换HttpService内部实现，对外契约不变。

### 2.2 路由表的实现方式

为了避免引入Express之类的依赖，自己实现了一个最小的路由表：

```js
addRoute('GET', '/movies/:id', handler)
```

`registerRoutes()` 在构造时把模式编译成 `RegExp`，请求到来后顺序遍历匹配，命中后把 `:id` 这类段捕获到 `params` 对象。这种最小实现完全够用，因为：

- 当前只需要 ≤ 50 个端点；
- 顺序明确：把 `/movies/search` 注册在 `/movies/:id` 之前，避免被通配段吞掉；
- URL编码的演员名（如 `%E6%B6%BC%E6%A3%AE%E7%8E%B2%E5%A4%A2`）会在 `matchRoute` 中自动 `decodeURIComponent`，便于中文/日文演员名传参。

### 2.3 统一响应规范

严格按照 `design/api-design.md` 的1.2/1.3章节落地：

```json
{
  "code": 0,
  "message": "OK",
  "data": { ... },
  "timestamp": 1717891200000
}
```

错误码集中在 `RESPONSE_CODE` 常量里，HTTP状态码通过 `HTTP_STATUS` 表映射，避免散落到每个控制器。

### 2.4 生命周期管理

`HttpService` 提供 `start()` / `stop()` / `restart()` / `isRunning()` / `getListenInfo()` 五个方法：

- `start()` 读取 `settings.http`，按 `enabled` 决定是否监听；
- `restart()` 在用户修改设置后由 `ipc-handlers.save-settings` 自动触发，实现热生效；
- `before-quit` 钩子里会调用 `stop()`，避免进程退出后端口仍被占用。

值得注意的实现细节：监听端口 `0` 时由操作系统随机分配，必须在 `listen` 回调里通过 `server.address().port` 读取真实端口写入 `listenInfo`——这一点在单元测试调试时踩到了坑，最终修复在 `HttpService.js:128-134`。

### 2.5 CORS 与 OPTIONS

考虑到移动端浏览器或Web客户端会跨域访问，默认对所有路径开启 CORS（`Access-Control-Allow-Origin: *`），并对 `OPTIONS` 预检请求直接返回 204，与设计文档的 1.1 章节一致。

### 2.6 请求体大小限制

`parseRequestBody` 设置 1 MB 上限，超过即销毁连接并抛错。这是为了避免恶意/异常客户端发送超大Body耗尽内存，符合CleanCode中"防御性编程"的实践。

### 2.7 与设置（SettingsService）的对接

在 `SettingsService` 中新增 `getHttpConfig()` / `setHttpConfig(config)` 两个对称方法，沿用项目里 `getProxyConfig` / `setProxyConfig` 的命名风格。`HardCodeService.getDefaultSettings()` 中加入 `http` 节作为默认配置，保证旧配置文件升级后自动获得新字段。

### 3.7 第七步：调试与修复

执行测试时遇到一个问题：所有端到端用例报 `connect ECONNREFUSED 127.0.0.1:80`。

排查：监听端口设置为 0 时，操作系统会随机分配；但代码里 `listenInfo` 仍然写入用户传入的 0，请求时就退化成默认 HTTP 端口 80 ——自然连不上。

修复方案：在 `listen` 回调里读取 `server.address().port` 作为真实端口（`HttpService.js:128-134`）。修复后 63/63 用例全部通过。

## 五、后续可扩展项

本次实现已覆盖设计文档中所有"读取类"GET端点。后续可继续按需补齐：

1. **写入类端点**：`PUT /movies/:id/rating`、`POST /boxes`、`PUT /boxes/:name`、`DELETE /boxes/:name`、`POST /history` 等；
2. **资源类端点**：海报 / 截图 / 演员照片 / 视频流（含Range支持）；
3. **缓存与ETag**：列表接口添加 `ETag` 与 `If-None-Match` 协商；
4. **鉴权**：移动端联网部署时建议在 `handleRequest` 入口处增加 `Authorization: Bearer <token>` 校验；
5. **限流**：高并发场景下可在 `handleRequest` 加入按 IP 的限速保护。

以上扩展点不需要重构现有架构，只需在 `registerRoutes` 中追加路由、在 `handleRequest` 中插入中间件即可。
